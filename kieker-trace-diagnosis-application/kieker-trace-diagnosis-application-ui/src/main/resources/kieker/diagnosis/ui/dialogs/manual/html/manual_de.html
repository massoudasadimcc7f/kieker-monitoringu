Einleitung

Kieker Trace Diagnosis ist eine Anwendung, die entwickelt wurde, um Kieker Monitoring Logs tabellarisch aufzubereiten und durchsuchbar zu machen. Hierbei wurde ein Fokus auf Speicherverbrauch und Performance gelegt, sodass auch größere Logs im Bereich von einigen Millionen Methodenaufrufen noch geöffnet werden können.

Anforderungen und Installation

Die folgenden Anforderungen sind notwendig, um Kieker Trace Diagnosis verwenden zu können.
  • Windows oder Linux
  • Java Runtime Environment 9 oder höher
Für die Verwendung des Werkzeuges wird das Oracle JDK empfohlen. Die Problembehandlung bei Verwendung des OpenJDK findet sich in Abschnitt OpenJDK. Der Speicherbedarf des Werkzeuges hängt von der Größe der Monitoring Logs ab. Als Richtlinie wird etwa das Zweifache der Größe des Monitoring Logs empfohlen.

Zur Installation muss lediglich das Archiv Kieker Trace Diagnosis-${version}-linux.tgz, beziehungsweise Kieker Trace Diagnosis-${version}-windows.zip, in einen beliebigen Ordner entpackt werden. Die Anwendung kann anschließend über das Startscript start.sh, beziehungsweise start.bat, im Ordner bin gestartet werden. Die Anwendung sollte in dem Verzeichnis, aus welchem das Startscript heraus aufgerufen wird, Schreibrechte besitzen.

Ansichten

Traces

Dieser Tab zeigt die Traces des Monitoring Logs einzeln an. Nicht rekonstruierbare Traces werden hier nicht angezeigt. Die Ansicht ermöglicht die Suche nach diversen Parametern, wobei der Filter standardmäßig nur innerhalb des obersten Methodenaufrufs sucht, d.h. in dem Wurzelknoten des Traces. Bei Bedarf kann mittels der Checkbox "Im gesamten Trace suchen" die Suche auf den gesamten Trace ausgeweitet werden. Ist die Checkbox "Mit regulären Ausdrücken suchen" aktiv, so kann in den Filtermasken "Host", "Klasse", "Methode" und "Exception" ein regulärer Ausdruck nach Standard-Java-Syntax verwendet werden.

Methodenaufrufe

Dieser Tab zeigt die Methodenaufrufe des Monitoring Logs einzeln an. Ignorierte Methoden und Methoden nicht rekonstruierter Traces werden hier nicht angezeigt. Die Ansicht ermöglicht die Suche nach diversen Parametern. Ist die Checkbox "Mit regulären Ausdrücken suchen" aktiv, so kann in den Filtermasken "Host", "Klasse", "Methode" und "Exception" ein regulärer Ausdruck nach Standard-Java-Syntax verwendet werden.

Aggregierte Methodenaufrufe

Dieser Tab zeigt die Methodenaufrufe des Monitoring Logs in aggregierter Form an. Aggregiert wird nach Host, Klasse, Methode und Exception, das heißt Methoden, bei denen alle vier Werte übereinstimmen werden zu einem aggregierten Methodenaufruf zusammengefasst. Ignorierte Methoden und Methoden nicht rekonstruierter Traces werden hier nicht angezeigt. Die Ansicht ermöglicht die Suche nach diversen Parametern. Ist die Checkbox "Mit regulären Ausdrücken suchen" aktiv, so kann in den Filtermasken "Host", "Klasse", "Methode" und "Exception" ein regulärer Ausdruck nach Standard-Java-Syntax verwendet werden.

Statistiken

Dieser Tab zeigt diverse Statistiken über den importierten Monitoring Log und die Anwendung an. Auch der aktuelle (Heap) Speicherbedarf der Anwendung kann hier abgelesen werden.

Einstellungen

Die Einstellungen erlauben die Konfiguration des Werkzeugs.

Unter dem Überpunkt "Darstellung" lassen sich Einstellungen vornehmen, die sich direkt auf die visuelle Repräsentation der Daten auswirkt. So kann die gewünschte Zeiteinheit oder die Darstellung von Methoden verändert werden. Die Checkbox "Nicht gemonitorte Zeit anzeigen" blendet zusätzlich im Trace die Zeit ein, welche in dem Methodenaufruf verbraucht wurde, aber aufgrund eines nicht vorhandenen Monitorings nicht genauer aufgeschlüsselt werden kann.

Mittels des Überpunktes "Methodenaufrufe in Traces zusammenfassen" lassen sich Methodenaufrufe in Traces zusammenfassen. Das ist dann nützlich, wenn ein Trace einige tausend Methodenaufrufe unter sich hat, von denen aber beispielsweise nur diejenigen interessant sind, welche die meiste Zeit verbraucht haben. Zur Auswahl stehen mehrere Aggregationsarten.

Monitoring Einstellungen

In den Monitoring Einstellungen kann das Monitoring für Kieker Trace Diagnosis eingeschaltet und konfiguriert werden. Das Monitoring ist im Allgemeinen nur für die Entwicklung von Kieker Trace Diagnosis oder für Lehrzwecke erforderlich. Die dabei entstehenden Logs können von dem Werkzeug ebenfalls eingelesen werden.

Problembehandlung

Verwendung von OpenJDK

Die Verwendung von OpenJDK anstelle des Oracle JDKs ist grundsätzlich möglich. Es kann allerdings notwendig sein, OpenJFX explizit zu installieren.

Unable to create file logs/Kieker Trace Diagnosis.log

Kommt es beim Start zu dieser Fehlermeldung, so hat Kieker Trace Diagnosis vermutlich in dem aktuellen Arbeitsverzeichnis keine Schreibrechte. Die Anwendung kann dann keine Log-Dateien schreiben, kann aber wie gehabt verwendet werden.

OutOfMemoryError

Kommt es bei der Verwendung des Werkzeugs zu einem OutOfMemoryError, so ist das Monitoring Log zu groß, um von Kieker Trace Diagnosis geladen zu werden. In diesem Fall sollte der Parameter -Xmx in dem Startscript auf einen höheren Wert gesetzt werden.
